# Roadmap: CallFlow v1.0.0 Stable

Status: roadmap toward stable release.

## Beta baseline

CallFlow `v0.3.0` is the frozen stable baseline for the v1.0 line. It replaces the earlier beta baseline as the foundation for the remaining stable-release work.

## Beta freeze rules

The beta is now frozen. Changes allowed on the beta line are strictly limited:

**Allowed:**

- Critical hotfixes (data loss, crash, security vulnerability)
- Security patches
- Installer fixes that prevent installation or first launch
- Minor documentation corrections (typos, broken links)

**Prohibited:**

- New features
- Large UI redesigns
- AI engine implementation
- Large refactors (e.g., app.js decomposition)
- New screens or workflow surfaces
- Dependency upgrades beyond security patches

## v1.0.0 objective

Deliver a stable, installable, polished Windows desktop app that fulfills the original CallFlow promise:

- A stable Windows installer that works reliably
- Final visual polish and UX coherence
- AI engine functional if implemented correctly, or explicitly disabled with clear messaging
- Final Windows QA pass with documented results
- Stable documentation aligned with the shipping product
- Honorable project closure: the repository can be archived or maintained with confidence

## In scope for v1.0

- Visual polish: colors, spacing, typography, icon consistency
- UX coherence: consistent button styles, modal patterns, feedback messages
- Graphical improvements: charts, stats layout, dashboard refinements
- AI engine: functional implementation with selected local context, or explicit placeholder with clear status
- Legal and privacy: final or stable-quality text replacing current drafts
- QA: full Windows validation pass for v1.0.0
- Release notes: complete v1.0.0 release documentation
- Changelog: updated through v1.0.0

## Out of scope for v1.0

- Cloud sync or remote data storage
- Mandatory backend server
- Mobile app (iOS, Android)
- Official Linux or macOS packaging and support
- Multi-user system or role-based access
- Large experimental features not validated in beta
- Auto-update mechanism (manual GitHub Releases workflow is sufficient)

## Versioning plan

| Version | Purpose |
| --- | --- |
| `v0.3.0` | Frozen stable baseline for v1.0 work |
| `1.0.0-rc.1` | First release candidate — all v1.0 scope implemented, QA in progress |
| `1.0.0-rc.2` | Second release candidate if rc.1 QA reveals issues |
| `1.0.0` | Stable release — QA passed, documentation finalized, installer published |

The `package.json` version will be bumped to `1.0.0-rc.1` when the first release candidate branch is created, not before.

## Exit criteria for v1.0.0

All of the following must be true before tagging `v1.0.0`:

- [ ] Windows installer builds and installs without errors
- [ ] Full onboarding flow completes on a clean Windows install
- [ ] All core workflows pass QA: calls, CRM copy, reports, reminders, stats, scripts, timers
- [ ] Backup export and import work correctly
- [ ] Local data erase works with confirmation modal and main-process token
- [ ] Dark and light themes render correctly across all views
- [ ] AI section is either functional with local context or clearly marked as disabled
- [ ] Legal text is stable-quality (not labeled as "draft" or "placeholder")
- [ ] README, release notes, changelog, and QA documents are coherent and up to date
- [ ] No known critical bugs or data-loss scenarios
- [ ] `npm run validate` passes (check, test, lint)
- [ ] SmartScreen behavior is documented (or installer is signed)
- [ ] Repository is presentable for portfolio, recruiters, and real users
